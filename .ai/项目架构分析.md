# `openclaw-a2a-gateway` 项目架构分析

## 1. 项目定位

这是一个 OpenClaw 插件，实现 A2A v0.3.0 网关能力。它既对外提供标准 A2A 接口，也把请求桥接到 OpenClaw agent 执行链，同时内建一些面向多 agent 网络的增强能力：路由、发现、熔断、重试、推送通知、审计和持久化任务。

从代码组织上看，它不是“单纯的 HTTP 服务”，而是：

1. 一个 OpenClaw 插件入口；
2. 一个 A2A 协议适配层；
3. 一个对等体（peer）通信客户端；
4. 一个带状态持久化、发现和韧性控制的网关。

## 2. 入口与运行时装配

`index.ts` 是最核心的文件，承担了几乎所有运行时装配职责：

- 解析插件配置：`parseConfig()`
- 创建核心组件：
  - `GatewayTelemetry`
  - `AuditLogger`
  - `PushNotificationStore`
  - `A2AClient`
  - `FileTaskStore`
  - `OpenClawAgentExecutor`
  - `QueueingAgentExecutor`
  - `PeerHealthManager`
  - `DnsDiscoveryManager`
  - `QuorumDiscoveryManager`
  - `MdnsResponder`
- 注册网关方法：
  - `a2a.send`
  - `a2a.metrics`
  - `a2a.audit`
  - `a2a.pushNotification.register`
  - `a2a.pushNotification.unregister`
- 注册工具：
  - `a2a_send_file`
- 提供 HTTP 路由：
  - Agent Card
  - JSON-RPC
  - REST
  - metrics
  - push notification 管理
- 启动 gRPC 服务（端口为 `server.port + 1`）
- 服务启动时恢复未终态任务，并启动 TTL 清理

因此，任何跨模块行为变更，通常都需要回到 `index.ts` 检查 wiring 是否完整。

## 3. 入站请求主链路

### 3.1 HTTP / JSON-RPC / REST / gRPC 入口

入站面由 `@a2a-js/sdk` 负责协议层处理：

- Agent Card：`agentCardHandler`
- JSON-RPC：`jsonRpcHandler`
- REST：`restHandler`
- gRPC：`grpcService`

这些处理器最终都共享同一个 `DefaultRequestHandler(agentCard, taskStore, executor)`。

### 3.2 持久化和执行

`DefaultRequestHandler` 的两个关键依赖是：

- `FileTaskStore`：任务持久化到磁盘 JSON 文件
- `QueueingAgentExecutor`：处理并发、排队、拒绝、取消和软限流

`QueueingAgentExecutor` 是真正把“任务生命周期”管理起来的地方，它负责：

- 超过并发上限时排队
- 队列满时显式拒绝
- 记录提交/开始/结束事件
- 在高负载下按 saturation 配置增加延迟
- 委托给底层 executor 执行

### 3.3 OpenClaw 执行桥

真正执行任务的是 `OpenClawAgentExecutor`。它的职责不是自己“推理”，而是把 A2A 请求桥接到 OpenClaw gateway agent RPC：

- 从 A2A 消息中提取文本
- 把 `FilePart` / `DataPart` 格式化为 agent 可理解的文本描述
- 维护上下文/session key
- 把 agent 输出重新组装为 A2A 任务状态和消息
- 把 `mediaUrl` / `mediaUrls` 转成 A2A `FilePart`

这意味着：

- 如果消息格式丢失、文件附件表示异常、上下文串联错误，通常要从 `src/executor.ts` 查起。
- 这里是 A2A 内容模型与 OpenClaw 内容模型之间最关键的转换层。

## 4. 出站 peer 调用链路

出站调用主要由两条入口触发：

- 网关方法 `a2a.send`
- 工具 `a2a_send_file`

二者最终都走 `src/client.ts` 的 `A2AClient.sendMessage()`。

`A2AClient` 的关键职责：

- 解析 peer 的 Agent Card URL
- 按 peer 配置附加 bearer/apiKey 鉴权
- 解析 Agent Card，收集可用 transports
- 构造标准 A2A outbound message
- 按 JSON-RPC -> REST -> gRPC 顺序尝试发送
- 对传输层错误执行 fallback
- 配合 `PeerHealthManager` 和 `peer-retry.ts` 执行重试/熔断
- 记录每个 transport 的近期成功率和延迟，用于自适应排序

相关模块分工：

- `client.ts`：发送主流程
- `transport-fallback.ts`：传输排序、自适应打分、重试型错误判定
- `peer-retry.ts`：指数退避重试包装
- `peer-health.ts`：健康状态和 circuit breaker

## 5. 发现、广告与网络韧性

这一组能力不是单个文件能看懂的，需要跨文件理解：

- `dns-discovery.ts`：DNS-SD peer 发现
- `quorum-discovery.ts`：基于 peer 密度的自适应轮询
- `dns-responder.ts`：mDNS 自我广播
- `peer-health.ts`：health check + circuit breaker
- `routing-rules.ts`：规则路由与 affinity 评分
- `saturation-model.ts`：软并发限制延迟模型

这些模块共同构成 README 里强调的 bio-inspired 能力。

需要注意的设计点：

- 这些能力默认都应是可选的。
- 未启用时，系统应退回标准 A2A/普通网关行为。
- 它们主要是增强“多 peer、大规模 agent 网络”的稳定性和效率。

## 6. 可观测性与持久化

### 6.1 Telemetry

`telemetry.ts` 聚合了：

- HTTP 请求计数
- 出站请求统计
- 任务队列/并发/完成状态
- peer 重试与健康状态
- 安全拒绝统计

`/a2a/metrics` 端点直接返回 telemetry snapshot。

### 6.2 Audit

`audit.ts` 负责 JSONL 审计日志，记录：

- 入站任务完成
- 出站调用结果
- 安全事件

网关方法 `a2a.audit` 可以 tail 最近的审计记录。

### 6.3 任务持久化与恢复

任务不是存在内存里，而是保存在磁盘：

- `task-store.ts`：读写任务文件
- `task-recovery.ts`：启动时恢复卡在非终态的任务
- `task-cleanup.ts`：按 TTL 定时清理终态任务

这说明任何任务状态流转修改，都要考虑：

1. 文件落盘格式是否兼容；
2. 启动恢复是否仍成立；
3. TTL 清理是否需要同步调整。

## 7. 安全模型

安全相关不是集中在一个“auth.ts”，而是分散在几个关键点：

- `index.ts`：HTTP/gRPC bearer token 校验
- `file-security.ts`：URI、MIME、大小、IP 范围安全校验
- `src/internal/security.ts`：内部扩展通道的安全机制（见 `src/internal/README.md`）

`file-security.ts` 是一个必须复用的关键模块，尤其是：

- 阻断私网/保留地址 SSRF
- 限制允许的 URI scheme
- 控制 MIME allowlist
- 控制文件大小
- 避免验证后再次解析 URL 导致的安全问题

## 8. `src/internal/*` 的边界

`src/internal/README.md` 已明确说明：这里不是标准 A2A 公共协议面，而是 OpenClaw 网关之间的内部扩展。

这意味着未来修改时要先判断目标属于哪一类：

- 标准 A2A surface：看 `index.ts`、`client.ts`、`executor.ts` 等主干文件
- OpenClaw mesh extension：看 `src/internal/*`

不要把内部扩展误当成标准协议的一部分。

## 9. 测试结构

测试采用 Node 原生测试栈：

- `node:test`
- `assert/strict`
- `tsx` 作为 TypeScript 测试加载器

关键测试文件：

- `tests/a2a-gateway.test.ts`：主功能、默认配置、桥接行为
- `tests/benchmark.test.ts`：bio-inspired 特性对比
- `tests/helpers.ts`：共享 harness、mock API、mock WebSocket、配置工厂

写新测试时应优先复用 `tests/helpers.ts`，而不是重新搭一套 mock。

## 10. 重要仓库约定

- `.ts` 文件中仍使用 `.js` import 路径。
- 配置项新增/变更必须同步修改：
  - `openclaw.plugin.json`
  - `index.ts` 中的 `parseConfig()`
- gRPC 是非致命能力；HTTP 面成功更优先。
- 静态 peers 和动态发现 peers 合并时，静态配置优先。
- 任务文件持久化是系统正常生命周期的一部分，不是测试或调试专用能力。

## 11. 实际开发时的快速定位建议

- **改配置**：先看 `openclaw.plugin.json` + `index.ts`
- **改入站任务执行**：先看 `executor.ts` + `queueing-executor.ts`
- **改对等体发送/传输回退**：先看 `client.ts` + `transport-fallback.ts`
- **改发现/自适应轮询**：先看 `dns-discovery.ts` + `quorum-discovery.ts`
- **改安全校验**：先看 `file-security.ts`
- **改任务持久化/恢复/清理**：一起看 `task-store.ts` + `task-recovery.ts` + `task-cleanup.ts`
- **改测试**：先看 `tests/helpers.ts`
